Codexに疑似Scratch音ゲー基盤を構築させるための調査報告
エグゼクティブサマリ
この案件では、Scratch 全体の完全再実装を目指すより、Scratch互換の最小サブセットを HTML/JS 上で実装し、その中間表現から Web 実行系と .sb3 生成系の両方へ落とす構成が最も実務的です。Scratch VM は各ターゲットごとにブロック AST を保持し、opcode、inputs、fields、next、parent、topLevel、x、y などの情報で実行状態と保存状態を表現します。したがって、Codex にはまずこの構造に寄せた Scratch互換 DSL / JSON を設計させ、その DSL から HTML/JS ランタイムと Scratch VM 互換 project.json を生成させるのが自然です。Scratch の標準ステージは 480×360 なので、内部座標系もここに固定し、ブラウザ表示は CSS 拡大で対応する方が、HTML 実装と .sb3 変換の双方で整合が取りやすくなります。 

音ゲーでは、描画側の時刻と音声側の時刻を分離して扱う必要があります。Web Audio の currentTime は音声再生スケジューリングに使える単調増加のハードウェア時間で、performance.now() は高解像度・単調増加の DOM 系タイムスタンプです。さらに AudioContext.getOutputTimestamp() を使うと、currentTime 系と performance.now() 系を対応付けられます。したがって、BGM と判定の真の基準時刻は AudioContext.currentTime、描画・入力ログ・HUD は performance.now()、両者の橋渡しは getOutputTimestamp() という三層で設計するのが最も堅牢です。 

ユーザーが提示した画像に含まれるカテゴリは、動き／見た目／音／イベント／制御／調べる／演算／変数／リスト／ブロック定義です。本報告では、これらを音ゲー向け最小互換カテゴリとして再整理し、Pen 相当は追加機能として別扱いにします。Scratch VM には motion、looks、sound、event、control、sensing、operator、data、procedures のコアブロック実装があり、拡張ブロックの引数型や opcode の定義方法は Scratch Editor の拡張仕様で説明されています。 

調査の基準と設計方針
今回の仕様化では、優先ソースを Scratch Foundation の公式リポジトリと Web Audio API の一次資料に置きました。具体的には、Scratch VM の README と serialization/sb3.js を .sb3 互換化の根拠に、各 scratch3_*.js をカテゴリ別ブロックの根拠に、Scratch GUI の costume-tab.jsx と sound-tab.jsx をアセット受理形式の根拠に、MDN と W3C を音声同期・高精度時間計測の根拠に使っています。Teachable Machine は本件の核心である Scratch 互換 runtime / .sb3 変換 / 音ゲー同期へ直接関与しないため、参照優先度は低いと判断しました。 

設計上の重要な判断は次の三つです。第一に、Scratch と完全同一の UI/エディタを再現しないことです。Scratch Editor の拡張仕様が示すように、Scratch のブロックは opcode、blockType、text、arguments を持つメタデータで定義されますが、本案件で Codex に作らせるべきなのは GUI エディタではなく、その背後の実行規約と保存規約です。第二に、内部のゲームロジックを Scratch 互換 DSL に集約することです。Scratch VM のシリアライザはターゲットごとに variables、lists、broadcasts、blocks、comments、costumes、sounds を保存し、ステージとスプライトで保存対象プロパティが異なります。第三に、音ゲー固有の時間同期だけは Scratch 風 API の裏側でネイティブ Web Audio を使うことです。これは Scratch 互換性よりも音ゲーの成立性を優先する判断です。 

実装対象は、音ゲー向け最小互換サブセットです。つまり「Scratch で一般に見慣れた構文と保存形を保ちながら、音ゲーに必要なブロックだけを優先実装する」方針です。具体的には、動き・見た目・音・イベント・制御・調べる・演算・変数・リスト・ブロック定義の十カテゴリを実装し、Pen 相当、クローン管理、デバッグ可視化、アセット管理、 .sb3 変換支援は追加機能として搭載します。この境界設定により、Codex が不用意に Canvas 直叩きのゲームコードや DOM 直操作へ逸脱するのを防げます。

Scratch VM の挙動で特に重要なのは、イベント hat は startHats で起動されること、broadcast and wait は起動したスレッドがランタイムから消えるまで待つこと、wait や glide はフレーム間で yield しながら進むこと、リストは 1-indexed であること、比較で数値文字列と数値が等価と見なされるケースがあることです。これらは Web 実装と .sb3 実装で差が出やすい部分なので、Codex への指示では必ず互換テスト対象に含めるべきです。 

アセット仕様は Scratch GUI と Scratch VM シリアライザに寄せます。Scratch GUI のアップロード受理形式は、コスチューム/背景で .svg, .png, .bmp, .jpg, .jpeg, .gif、サウンドで .wav, .mp3 です。Scratch VM の .sb3 シリアライザは、コスチュームに name、bitmapResolution、dataFormat、assetId、md5ext、rotationCenterX、rotationCenterY を、サウンドに name、assetId、dataFormat、format、rate、sampleCount、md5ext を保存します。ステージ保存時には tempo、videoTransparency、videoState、textToSpeechLanguage が、スプライト保存時には visible、x、y、size、direction、draggable、rotationStyle が保存対象です。

推奨アセット戦略としては、背景は backdrop、ノーツと判定演出は sprite、デバッグ線や一時的ガイドは Pen 相当に分けるのが管理しやすいです。Scratch 3 系ではビットマップ取り込み時の bitmapResolution を 2 倍系で扱う方針が議論されており、少なくとも 480×360 内部座標に対して等倍で破綻しない設計にしておくべきです。 

音ゲー固有仕様と Scratch互換 DSL
音ゲーでは、譜面データ、ノーツ生成、判定ウィンドウ、オフセット調整、音声同期基盤を最初から明示仕様化しておく必要があります。currentTime は音声スケジューリングの主時計、performance.now() は描画・ログ・入力時計、getOutputTimestamp() は両者の対応付けに使います。これを前提に、譜面は「音声時刻基準」で保存し、描画だけを DOM/Canvas 側へ変換する設計が安全です。

実装順とテストとデバッグ
実装順は、ブロック互換性の土台と音ゲー成立条件を交互に詰めるのがよいです。最短ルートは、Stage / Sprite / Variable / List / Broadcast / Thread のランタイム骨格、次に SoundBridge と ChartLoader、その後に JudgeSystem、Clone、SB3Generator です。Scratch VM はターゲット単位で blocks/state を持ち、イベント hat と制御ブロックで実行を駆動するので、Codex にはまず イベント駆動 thread ランタイムを作らせるべきです。
